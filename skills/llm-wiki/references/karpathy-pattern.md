# Karpathy LLM Wiki 模式 — 原始参考

来源：https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f

## 核心洞见

"Wiki 是一个持久的、复利的产物。知识被编译一次后保持更新，而非每次查询时重新推导。"

人类策划来源并提出问题；LLM 维护知识系统。Obsidian 成为 IDE，LLM 成为程序员，Wiki 成为代码库。

## 为何优于 RAG

传统 RAG 在每次查询时重新发现知识——它搜索原始来源、拉取相关分块、从零综合答案。LLM Wiki 将知识一次性编译为维护好的页面，因此查询命中预综合的、交叉引用的内容。

## 关键操作

| 操作 | 做什么 | 何时使用 |
|---|---|---|
| **摄取（Ingest）** | 阅读新来源，提取关键信息，更新 10-15 个 wiki 页面，维护一致性 | 当新文档到达时 |
| **查询（Query）** | 针对已编译的 wiki 回答问题并附引用 | 当用户提问时 |
| **检查（Lint）** | 识别矛盾、孤立页面、过时论断、缺失交叉引用 | 定期维护 |

## 推荐工具（Karpathy 原始模式中）

- **Obsidian** — 浏览和探索 wiki 的 IDE
- **Web Clipper** — 将文章转为 markdown 的浏览器扩展
- **Marp** — 基于 markdown 的幻灯片，内容来自 wiki
- **Dataview** — 用于查询页面元数据的 Obsidian 插件
- **qmd** — 具有 BM25/向量混合搜索的本地搜索引擎

## 本分支的适配（思源后端）

本分支保留了 Karpathy 模式的每个概念原语（编译而非检索、持久复利产物、摄取/查询/检查作为核心操作），但替换了存储和工具层：

- **Obsidian → 思源笔记。** 思源本身就是 IDE，桌面客户端运行后可通过 `http://localhost:6806` 访问。Wiki 文档是思源中一个命名笔记本内的文档（`SIYUAN_NOTEBOOK_ID` + `SIYUAN_NOTEBOOK_NAME`）。
- **文件系统 markdown → 块索引。** 页面仍以 markdown 正文序列化（使导出和迁移保持合理），但规范的查询路径是针对思源块索引的 `siyuan-sisyphus search query_sql`，从每个文档的 `ial` 列投射 `custom-*` 属性。仅当索引无法回答时才读取页面正文。
- **Dataview → `query_sql` +（可选）数据库块。** 针对块索引的 SQL 查询处理大多数元数据驱动的视图；思源 3.0+ 的数据库块为希望在思源客户端内进行筛选/排序/分组的用户提供了交互编辑界面。
- **qmd → `search fulltext`。** 思源内置了服务器维护的 BM25 全文索引，覆盖相同的检索需求；无需单独的嵌入管道。
- **Web Clipper → `ingest-url`。** 本分支中的一个技能获取 URL、蒸馏其内容，并通过 `siyuan-sisyphus fs write` 写入页面，而非将 markdown 文件放入文件夹。
- **反向链接（Obsidian 的全库 grep）→ `search get_backlinks`。** 服务器维护，同时跟踪 wikilink 引用和纯文本提及，每页 O(deg) 可查询。

Karpathy 的原则（"编译而非检索"；"Wiki 是一个编程产物"；"链接是骨架，而非装饰"）仍驱动这里的每个技能——只是底层基质变了。

## 应用场景

- 个人追踪（目标、心理、自我提升）
- 研究（在数周/数月中建立全面理解）
- 书籍标注（带有角色、主题、情节关联的伴随 wiki）
- 团队/商业（从 Slack 线程、会议记录构建 wiki）
- 尽职调查、竞争分析、旅行规划

## 值得了解的社区扩展

- **来源追踪** — 记录哪些来源文件产生了每个论断，通过内容哈希检测过时
- **层级继承** — 父子页面关系，而非扁平索引
- **决策记录** — 捕获 wiki 为何演进，而不仅是变更了什么
- **双层 LLM** — 本地模型处理敏感数据，云端处理其余
- **图数据库** — 类型化本体，而非 markdown 链接
