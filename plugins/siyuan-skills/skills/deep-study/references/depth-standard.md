# Depth Standard — L1-L4 深度标准与面试追问链

定义 deep-study 的深度判定标准，使"深度"可检验而非主观。每个 concept 页面必须达到 L4 深度。

## L1-L4 深度层级

| 层级 | 判定标准 | 达标标志 | 写入位置 |
|---|---|---|---|
| L1 定义 | 能用一句话说清"X 是什么" + 列出 2-3 个核心特征 | Flashcards L1 填空题能独立回想 | `## Flashcards` L1 段（`custom-card-type=cloze`）+ 定义段 |
| L2 原理 | 能讲清"X 怎么工作"的机制 / 流程，含关键数据结构和调用链 | Flashcards L2 问答题能答出关键步骤 | `## Flashcards` L2 段（`custom-card-type=qa`）+ `## How It Works` |
| L3 动机 | 能回答"为什么这样设计" + 列出被放弃的替代方案及权衡 | Flashcards L3 单选题能选对并理解选项差异 | `## Flashcards` L3 段（`custom-card-type=single-choice`）+ `## Why This Way` |
| L4 实战 | 能应对面试官追问 + 知道生产踩坑 + 有实践方案 | Flashcards L4 多选题 + Interview Questions + Common Pitfalls + Practice Ideas 预填 | `## Flashcards` L4 段（`custom-card-type=multi-choice`）+ 深入段 |

## 量化深度标准

判定标准不能停留在"能讲清""能回答"这种主观描述。以下是可度量的硬性要求：

### L1 定义段（≥ 200 字）

- 一句话定义（X 是什么 + 用在哪里）
- 3-5 个核心特征（不是形容词堆砌，是可验证的技术事实）
- L1 填空题：挖空核心术语，每张挖 1-3 处

### L2 原理段 How It Works（≥ 800 字）

- 机制 / 流程的完整描述（不是概括性的"从 A 到 B 到 C"，而是每步做了什么）
- **至少 1 处源码引用**：函数签名 + 文件名:行号 + 核心代码片段（≤ 20 行）
- **至少 3 个量化数据**：微秒数、内存比例、版本号、阈值等
- **至少 1 个数据结构说明**：关键 struct / 队列 / 缓存的字段和作用
- L2 问答题：问题聚焦机制关键步骤，答案含上述量化数据

### L3 动机段 Why This Way（≥ 500 字）

- "为什么这样设计"的核心原因（不是一句话，是含上下文的论证）
- **至少 2 个被放弃的替代方案**，每个含：替代方案是什么、为什么放弃、权衡点
- L3 单选题：4 个选项须是有道理的干扰项（不是明显荒谬的选项），正确选项 + 3 个"看起来对但错了"的干扰项

### L4 实战段（深入段）

#### Common Pitfalls（≥ 3 条，每条 ≥ 100 字）

每条含：**现象**（具体场景，不是"性能差"这种模糊描述）、**原因**（根因分析，追溯到源码或运行时行为）、**解法**（具体命令 / 配置 / 代码改动，不是"优化一下"）。

#### Interview Questions（≥ 6 个，含 L4 级追问）

- 至少 1 个 L4 级追问能区分"背了概念"和"做过"——通常是生产环境的边界条件、参数调优、故障排查
- 每个问题附答题方向（切入思路 + 关键点 + 易踩的坑），不是完整答案
- 追问须有递进性——后一个问题建立在前一个问题的答案上

#### Practice Ideas（≥ 2 个，含可运行代码）

- 每个含具体代码示例或架构设计（不是"建议试试 X"这种空泛建议）
- 须可落地——给读者一个起点：具体的命令、代码片段或项目骨架

## 闪卡质量标准

### 单选题干扰项设计

错误选项不是随机编造的，而是基于常见误解设计：

- 正确选项：真正正确的答案
- 干扰项 1：基于最常见的误解（如"引入分代 GC"——Go 确实没有分代，但 1.8 不是为了引入分代）
- 干扰项 2：技术上正确但不是主要解决的问题（如"压缩堆内存"——Go GC 确实不压缩，但 1.8 的改动不是为了压缩）
- 干扰项 3：看起来合理但完全错误（如"启用引用计数"——Go 从未用过引用计数）

### 多选题设计

- 正确选项 2-3 个（不是全选或单选）
- 错误选项基于常见误区（如"让变量逃逸到堆上"——看起来是优化但实际增加 GC 压力）
- 解析段须解释每个选项为什么对/错

## 面试追问链设计方法

追问链是 deep-study 的核心产出之一（见 `deliverable.md`），设计方法：

1. **从 L1 起步**：用最基础的定义问题开场，确认对方懂概念。
2. **逐层递进**：L1→L2→L3→L4，每层 1-3 个问题，后一层建立在前一层答案上。
3. **答题方向非完整答案**：每个问题附"答题方向"——答题的切入思路、关键点、易踩的坑，不写完整答案（完整答案在 concept 正文）。
4. **L4 聚焦实战**：最后一层必须是生产环境问题（踩坑 / 调优 / 选型），这是区分"背了概念"和"真正做过"的关键。
5. **6-10 个问题**：太少覆盖不了深度，太多冗余。典型 7-8 个。
6. **至少 1 个"杀手问题"**：能瞬间区分"背了概念"和"做过"的问题。通常是边界条件、参数精确值、或"为什么不能做 X"的反直觉问题。

## 示例追问链（主题：Go 垃圾收集）

| # | 层级 | 问题 | 答题方向 |
|---|---|---|---|
| 1 | L1 | Go 的 GC 用的是什么算法？ | 三色标记 + 并发标记清除，非分代非压缩 |
| 2 | L2 | 三色标记具体怎么工作？白 / 灰 / 黑的含义和流转？ | 白=未访问、灰=已访问但子节点未扫、黑=完成；从灰队列出发扫子节点 |
| 3 | L2 | 并发标记怎么处理 GC 期间对象引用变化？ | 混合写屏障（Dijkstra+Yuasa），黑对象引用白对象时标灰 |
| 4 | L3 | 为什么 Go 不用分代 GC？ | Go 栈上分配 + 逃逸分析减少堆压力；分代收益小且增加复杂度；权衡 simplicity |
| 5 | L3 | 写屏障为什么从 Dijkstra 切到混合写屏障？ | 1.8 前栈重扫描成本高；混合写屏障消除栈重扫描，降低 STW |
| 6 | L4 | 生产中遇到 STW 尖刺怎么排查？ | GODEBUG=gctrace=1 看 GC 频率；pprof 找分配热点；GOGC 调阈值或用 GOMEMLIMIT |
| 7 | L4 | **Mark Assist 在什么条件下被触发？assist ratio 怎么算？它如何影响 P99？** | 分配速率超过 GC 回收速率时触发；assist ratio = 期望堆大小 / 已标记大小；高分配率服务 P99 可能恶化 20-30% |

第 7 题是"杀手问题"——没做过 Go GC 调优的人答不出 assist ratio 的影响。

## 深度不达标的处理

研究素材不足以支撑 L4 时（如新主题缺乏实战案例）：

1. 仍创建 concept 页面，L1-L4 闪卡正常注册。
2. 深入段预填能推断的内容，无法推断的留空并加 `> [!CAUTION] 此段待深化，研究素材不足`。
3. `custom-status=draft`，日志注明"L4 待深化：<具体缺口>"。
4. 在 `## Practice Ideas` 写"建议通过 <具体实践> 补充实战经验"。

不伪造实战内容——L4 的价值在于真实性，杜撰的踩坑比留空更有害。
