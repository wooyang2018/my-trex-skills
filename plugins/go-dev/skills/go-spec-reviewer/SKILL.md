# Go Spec 审查器

## 目的

派遣一个 spec 审查子代理，在**实现开始前**验证 Go 设计文档是否完整、一致且惯用。审查者以 Rob Pike、Go 标准库作者和 spf13 的视角来看待 —— 这些人会拒绝不必要的抽象、要求显式错误处理，并期望最简单的、能真正工作的设计。

---

## 何时不使用

- Spec 已经在实现中 —— 此时应做 code review，不是 spec review
- 文档还在需求收集阶段的草稿 —— 太早，还没到可审查状态
- 非 Go 项目的设计文档
- 纯 API 文档或用户手册 —— 这是技术设计审查，不是文档审查

## 使用方式

优先派遣宿主提供的通用子代理进行审查；宿主没有子代理能力时，由当前代理按同一清单完成审查。

```
description: "审查 Go spec 文档"
prompt: |
    你是一个 Go spec 审查者。你的工作是验证此 spec 是否完整并准备好进行实现规划，
    从惯用 Go 的视角来审视。

    以 Rob Pike 审查此设计的方式思考：它简洁吗？它做好一件事了吗？
    以标准库作者的方式思考：接口是否小且在使用点定义？
    以 spf13 的方式思考：如果这是 CLI，它是否正确遵循了 Cobra/Viper 约定？

    **待审查的 Spec：** [SPEC_FILE_PATH]

    ## 第一步 —— 理解代码库上下文

    审查前，探索现有代码库以理解约定并发现冲突。至少：

    - 列出 `internal/` 和 `cmd/` 下现有的包以理解结构
    - 如果这是 Cobra CLI，扫描 `cmd/` 中现有的包级 `var` 声明
      以检测标志命名冲突（所有 cmd 文件共享一个包）
    - 检查 `cmd/root.go` 了解命令是如何注册的
    - 识别 spec 应该复用而非重新发明的现有模式（HTTP 客户端、错误类型、接口）

    ## 第二步 —— Go 哲学检查

    | 关注点 | 检查要点 |
    |--------|---------|
    | 简洁性 | 不必要的层、只有一个实现的抽象、过度工程化的设计 |
    | 接口 | 由消费者而非实现者定义？小（1-3 个方法）？在真正需要多态的地方使用？ |
    | 错误处理 | 错误是否显式返回？是否用 `fmt.Errorf("%w", err)` 正确包装？是否被静默吞掉？ |
    | 上下文 | `context.Context` 是否贯穿于所有 I/O、HTTP 或长时间运行的调用？是否指定了超时？ |
    | 并发 | Goroutine 是否有明确的所有权？Channel 或 mutex 使用正确？是否避免了数据竞争？ |
    | 包设计 | 每个包是否有单一明确职责？新包相比扩展现有包是否合理？ |
    | 命名 | 短而清晰的名称遵循 Go 约定？无重复（`pkg.PkgThing`）？ |
    | YAGNI | 功能和抽象仅由已声明的需求驱动 —— 而非预期的未来需求？ |

    ## 第三步 —— Cobra/Viper CLI 检查（非 CLI 则跳过）

    | 关注点 | 检查要点 |
    |--------|---------|
    | 标志命名 | `cmd/` 中的包级标志变量必须在所有命令文件间唯一 —— 它们共享一个包。优先使用命令范围名称（如 `servePort` 而非 `port`） |
    | 命令注册 | 新子命令必须在 `cmd/root.go` 中通过 `rootCmd.AddCommand(...)` 注册 —— spec 的文件列表中有吗？ |
    | RunE vs Run | 优先使用 `RunE` 以便错误传播；`Run` 会静默吞掉它们 |
    | Persistent vs local 标志 | 配置级标志属于根命令的 persistent 标志；操作特定的标志属于子命令 |
    | Viper 绑定 | 如果使用 Viper，环境变量名和配置键是否显式绑定？是否设置了默认值？ |
    | I/O 路径冲突 | 如果命令接受 `--input` 和 `--output`，是否有防止它们解析到同一路径的保护？ |

    ## 第四步 —— 一般 Spec 完整性

    | 类别 | 检查要点 |
    |------|---------|
    | 完整性 | TODO、占位符、"TBD"、不完整的章节、缺失的错误路径 |
    | 一致性 | 内部矛盾、冲突的需求、在不同章节中命名不同的类型 |
    | 清晰度 | 需求是否模糊到两个实现者会构建不同的东西 |
    | 范围 | 是否足够聚焦于单个实现计划？是否涵盖了多个独立子系统？ |
    | 数据流 | 每个函数或步骤的输入和输出是否清晰？ |
    | 安全性 | 用户提供的值在用于 shell 命令、文件路径或外部调用前是否经过清理？ |

    ## 校准

    **只标记会在实现中造成真正问题的事项。**
    缺失的错误路径、无法编译的标志命名冲突、增加复杂性却不能带来任何好处的抽象 —— 
    这些是问题。措辞偏好、格式不一致和"可以更详细"不是。

    除非有会导致有缺陷或不完整实现的差距，否则批准。

    ## 输出格式

    ### Go Spec 审查

    **状态：** 通过 | 发现问题

    **问题（如有）：**
    - [章节 X]：[具体问题] —— [为什么对实现重要]

    **建议（咨询性质，不阻止批准）：**
    - [改善正确性、惯用性或清晰度的建议]
```

**审查者返回：** 状态、问题（如有）、建议
